Zabbix agent 在 Windows 上安装部署

下载与解压

  • 下载地址：http://www.zabbix.com/downloads/3.0.0/zabbix_agents_3.0.0.win.zip

  • 解压后 conf 目录存放着 agent 配置文件， bin 目录存放windows下32和64位安装程序

配置与安装

  • 找到 conf 目录下的配置文件 zabbix_agentd.win.conf，修改Server、ServerActive、Hostname这三个参数，具体配置参见 Zabbix Agent Installing And Configuration

  • 安装 agent

  • 启动 agent

测试

  • 确认防火墙已开启 zabbix_agent 的通讯端口，默认为 tcp 10050

  • 查看 agent 日志文件确认启动正常

  • 在 zabbix 服务端使用 zabbix_get 工具进行测试

Zabbix 事件

1. Zabbix 事件通知

1.1 概述

我们花费大量时间去配置 item、tirgger 都是为了发送报警做准备的，什么是事件通知？简单说就是故障发生了，zabbix 会发邮件或者短信给你，告诉你服务器的一些状况。

1.2 通知条件

发送通知，需要有如下步骤：

  • 定义一个通知介质

这个介质包含 email、shell 脚本、sms 短信网关等。zabbix 将需要发送的内容传递给这些介质，然后这些介质把需要发送的内容给到对应的终端。

  • 配置 action

Action 由 conditions (条件)和 operations (操作)组成。当满足指定的条件，然后执行操作。这就是一个 action。

1.3 事件通知配置

  • 点击 Configuration -> Actions

  • 点击右上角 Create action

此处我们选择右上角 Event source 为 Triggers

  • 配置 Action 标签页
项目 定义
Name action 名称
Default subject 报警通知的主题，一般是邮件主题
Default message 通知内容，一般就是邮件内容
Recovery message 勾上它，监控项恢复后也会收到通知
  • 配置 Conditions 标签页
项目 定义
Type of calculation 各条件之间的关系
Conditions 条件列表
New condition 此处可以添加更多的条件
  • 配置 Operations 标签页
项目 定义
Default operation step duration 默认配置，表示这次发送报警后，下一次发送是在多少秒之后
Steps 发送次数
Step duration 发送间隔，如果设置为0，使用默认值
Operation type 发送介质
Send to User groups 发送到哪个用户组
Send to Users 发送给哪个用户

2. Zabbix 事件来源

2.1 概述

在 trigger 的文章内，我们已经讲到过事件，zabbix 事件我们这里介绍三种，分别为：触发器事件、发现事件、内部事件。

2.2 Zabbix 内部事件

内部事件是从 zabbix 2.2 版本开始引入，主要目的是为了可以通知管理员 zabbix 的某些 item 不可用。

  • 监控项 item 状态从 normal 变为 unsupported，或者从 unsupported 变为 normal

  • low-level 发现规则状态从 normal 变为 unsupported，或者从 unsupported 变为 normal

  • 触发器状态从 normal 变为 unknown，或者从 unknown 转为 normal

属性 描述
normal 正常
unsupported 不正常
unknown 未知

2.3 Zabbix 发现事件

在我们配置玩 zabbix 网络发现规则之后，zabbix 会定期按照这个规则去扫描 IP 段，一旦有发现主机和服务，变生成一个事件。事件如下列表：

事件 描述
Service Up 每当 zabbix 检测到 active service
Service Down 每当 zabbix 检测不到 service
Host Up 目标IP，至少有一个 service 是 UP
Host Down 所有 services 都没有响应
Service Discovered service 在维护时间之后恢复或者第一次被发现
Service Lost service 在 up 之后丢失了
Host Discovered 主机在维护时间之后恢复或者第一次被发现
Host Lost 主机在 up 之后丢失

2.4 Zabbix 触发器事件

触发器在 zabbix 事件当中是最重要也是最常用的事件（例如定义action等）。每当触发器状态发生变化都会产生一个包含详细状态信息的触发器事件：例如事件什么时候发送的以及当前状态是什么。

Zabbix 触发器表达式详解

概述

触发器中的表达式使用很灵活，我们可以据此创建一个复杂的逻辑测试监控，触发器表达式形式如下：

{主机:key.函数(参数)}<表达式>常数

Function 参数

大多数数值 functions 可以使用秒来作为参数。你可以使用”#”来表示它有不同的含义

函数 描述
sum(600) 600 秒内的总和
sum(#5) 最新 5 个值的和

触发器示例

说明：

参数 说明
mail.geak.cn host 名称
system.cpu.load[percpu.avg1] item 值，一分钟内 cpu 平均负载值负载值
avg(5m) 5 分钟内的平均数据
>5 大于5

如上所述，mail.geak.cn 这个数据的监控项，5 分钟的平均 cpu 的 1 分钟负载大于 5，表达式会返回 true，触发器状态就会改变为 “problem”了。

Zabbix 单位符号 Unit symbols

在 zabbix 里面，我们不需要使用大数字，例如可以不使用 86400 来表示一天，这个数字不容易理解也容易出错，用什么办法可以解决呢？我们使用单位来简化，例如简化 zabbix 触发器表达式或者 item key。所以，我们可以使用 1d 来取代 86400 ，’d’ 就是单位后缀，表示天。

Zabbix item key 详解

zabbix的items用来从主机获取数据，通常情况下，我们称item为监控项。例如我们服务器需要监控它的CPU负载，那么实现这个方法的东西就叫做item。

监控项需要获取cpu信息，则需要一个对应的监控key：system.cpu.load。如果是获取网卡流量，那么获取网卡这个监控项需要key：net.if.in或者net.if.out。

一般情况下key要与参数结合起来使用，例如获取5分钟的负载情况：system.cpu.load[avg5]，avg5是对应的参数。
