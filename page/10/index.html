客户端请求 WEB 流程

1. 客户端发送请求

  1. 客户端查询需要访问的域名的 A 记录。
  2. 客户端（通过浏览器）和 Web 服务器建立 TCP 连接。
  3. 连接建立以后，客户端向 Web 服务器发送访问请求（如get）。根据 HTTP 协议，该请求中包含了客户端的浏览器类型、请求的URL和HTTP版本等一系列信息。

2. 服务器解析请求

  1. Web 服务器对请求按照 HTTP 协议进行解码来确定进一步的动作，涉及的内容有三点：方法（get）、和浏览器使用的协议。
  2. 服务器现在知道了它应该使用什么协议返回哪个文件给客户。信息是经过与请求相同的连接发出的，所以服务器不需要创建新的连接。

3. 读取其它信息（非必须步骤）

Web 服务器根据需要读取请求的其它部分。在 HTTP/1.1 下，客户端还应给服务器提供关于它的一些信息。元信息（metainformation）可用来描述浏览器及其能力，以使服务器能据此确定如何返回应答。

4. 完成请求的动作

如现在没有错误出现，WWW 服务器将执行请求所要求的动作。

  1. 如果成功：文件被发送出去。首先，WEB 服务器发送一个状态码及一些描述信息。既然文件已经找到，则发送状态码200，表示一切都 OK，文档随后发出，因为发送的信息是 HTML 文档，所以 Content-type 取值为 text/html。服务器软件的标识及文件的时间属性也被包含在头域中。
  2. 如果失败：返回错误指示。如果请求的文件没有找到或找到但无法读取，则请求无法满足。这时将返回不同于200的状态码。

5. 关闭文件和网络连接，结束会话

当文件已被发出或错误已发出后，WEB 服务器结束整个会话。它关闭打开的被请求文件，关闭网络端口从而结束网络连接。有关其它工作则是由客户端来完整的，包括接收数据，并以用户可读的方式呈现出来。这些与服务器无关。

Redis

Redis 是一种高级的 key-value 数据库。它跟 memcached 类似，不过数据可以持久化，而且支持的数据类型很丰富。有字符串、链表、哈希、集合和有序集合 5 种。支持在服务器端计算集合的并、交和补集等，还支持多种排序功能。所以 Redis 也可以被看成是一个数据结构服务器。Redis 的所有数据都是保存在内存中，然后不定期的通过异步方式保存到磁盘上（这称为“半持久化模式”）；也可以把每一次数据裱画都写入到一个 append only file (aof) 里面（这称为“全持久化模式”）。

Nginx Location Configuration

语法规则：location [=|~|~*|^~] /url/ {…}

  • = 表示精确匹配

  • ^~ 表示 url 以某个常规字符串开头，理解为匹配 url 路径即可

  • ~ 表示区分大小写的正则匹配

  • ~* 表示不区分大小写的正则匹配

  • !~和!~* 分别为区分大小写不匹配和不区分大小写的不匹配

  • / 通用匹配，任何请求都回匹配到

多个 location 配置的情况下匹配顺序为：首先匹配 =，其次匹配 ^~，其次按文件中顺序的正则匹配，最后是交给 / 通用匹配。当有匹配成功的时候，停止匹配，按当前匹配规则处理请求。

例子

那么产生的效果如下：

  1. 访问根目录/，比如 http://localhost/ 将匹配规则 ruleA

  2. 访问 http://localhost/static/a.html 将匹配规则 ruleC

  3. 访问 http://localhost/login 将匹配规则 ruleB

  4. 访问 http://localhost/a.gif、http://localhost/b.png 将匹配规则 ruleD 和 ruleE，但是规则 ruleD 优先，ruleE 不起作用

  5. 访问 http://localhost/static/c.png 则优先匹配规则 ruleC

  6. 访问 http://localhost/a.PNG 则匹配规则 ruleE，不匹配规则 ruleD，因为规则 ruleE 不区分大小写

  7. 访问 http://localhost/a.xhtml 不会匹配规则 ruleF 和 ruleG，http://localhost/a.XHTML 不会匹配规则 ruleG。规则 ruleF 和 ruleG 属于排除法，符合匹配规则但是不会匹配。

  8. 访问 http://localhost/category/id/1111 则最终匹配到规则 ruleH，因为以上规则都不进行匹配

所以在实际使用中，个人觉得至少有三个匹配规则定义，如下：

  • 直接匹配网站根，通过域名访问网站首页比较频繁，使用这个会加速处理
  • 处理静态文件请求，这是 nginx 作为 http 服务器的强项
  • 转发动态请求到后端应用服务器
