什么是 Web 框架?

Web 应用框架，或者简单的说是 “Web 框架”，其实是建立 Web 应用的一种方式。从简单的博客系统到复杂的 AJAX 应用，Web 上每个页面都是通过代码来产生的。很多人热衷于学习 Web 框架技术，但并不理解什么是 Web 框架，或者它们是如何工作的。这里，我们将探讨反复被忽略的 Web 框架基础的话题。阅读完这篇文章，你应该对什么是 Web 框架以及它们为什么会存在有更深的认识。这会让你学习一个新的 Web 框架变的简单的多，还会让你使用不同的框架的时候作出明智的选择。

MikroTik

MikroTik 基于 IPv6 的 6to4 通道

6to4 采用特殊的 IPv6 地址使在 IPv4 海洋中的 IPv6 孤岛能相互连接。此时 IPv6 的出口路由器与其它的 IPv6 域建立隧道连接。IPv4 隧道的末端可从 IPv6 域的地址前缀中自动提取，通过这个机制，站点能够配置 IPv6 而不需要向注册结构申请 IPv6 地址空间。（也就是说，即使没有 IPv6 地址，通过 6to4 通道，别人也可以通过你申请的 IPv6 的访问，但是这个 IPv6 地址为动态）

sp_who 返回包括以下信息的结果集

列 含义
spid 会话ID
ecid 与特定会话ID相关的给定线程的执行上下文ID
status 进程状态
loginame 与特定进程相关联的登录名
hostname 每个进程的主机或计算机名
blk 如果存在阻塞进程，则是阻塞进程的会话ID，否则该列为零
dbname 进程使用的数据库
cmd 为该进程执行的数据库引擎命令
request_id 特定会话中运行的请求的ID

status：可以有的值:

. dormant: SQL Server 正在重置会话 . running: 会话正在运行一个或多个批处理，多个活动的结果集（MARS）启用后会话可以运行多个批处理
. background: 会话正在运行一个后台任务，例如死锁检测 . rollback: 会话正在处理事物回滚
. pending: 会话正在等待工作线程变为可用 . runnable: 会话的任务在等待获取 Scheduler 来运行的可执行队列中
. spinloop: 会话的任务正在等待调节锁变为可用 . suspended: 会话正在等待事件完成

客户端请求 WEB 流程

1. 客户端发送请求

  1. 客户端查询需要访问的域名的 A 记录。
  2. 客户端（通过浏览器）和 Web 服务器建立 TCP 连接。
  3. 连接建立以后，客户端向 Web 服务器发送访问请求（如get）。根据 HTTP 协议，该请求中包含了客户端的浏览器类型、请求的URL和HTTP版本等一系列信息。

2. 服务器解析请求

  1. Web 服务器对请求按照 HTTP 协议进行解码来确定进一步的动作，涉及的内容有三点：方法（get）、和浏览器使用的协议。
  2. 服务器现在知道了它应该使用什么协议返回哪个文件给客户。信息是经过与请求相同的连接发出的，所以服务器不需要创建新的连接。

3. 读取其它信息（非必须步骤）

Web 服务器根据需要读取请求的其它部分。在 HTTP/1.1 下，客户端还应给服务器提供关于它的一些信息。元信息（metainformation）可用来描述浏览器及其能力，以使服务器能据此确定如何返回应答。

4. 完成请求的动作

如现在没有错误出现，WWW 服务器将执行请求所要求的动作。

  1. 如果成功：文件被发送出去。首先，WEB 服务器发送一个状态码及一些描述信息。既然文件已经找到，则发送状态码200，表示一切都 OK，文档随后发出，因为发送的信息是 HTML 文档，所以 Content-type 取值为 text/html。服务器软件的标识及文件的时间属性也被包含在头域中。
  2. 如果失败：返回错误指示。如果请求的文件没有找到或找到但无法读取，则请求无法满足。这时将返回不同于200的状态码。

5. 关闭文件和网络连接，结束会话

当文件已被发出或错误已发出后，WEB 服务器结束整个会话。它关闭打开的被请求文件，关闭网络端口从而结束网络连接。有关其它工作则是由客户端来完整的，包括接收数据，并以用户可读的方式呈现出来。这些与服务器无关。
